II.BACKGROUND  /  项目背景

02Three persistent failures

医疗数据共享的三个核心痛点

01
Data Silos
数据孤岛
不同医院的病历系统彼此隔离，跨院调阅困难，重复检查浪费医疗资源。
02
Trust Deficit
信任缺失
中心化存储易被篡改且难以独立追溯，授权审批也缺乏不可抵赖的证据。
03
Control Imbalance
控制权失衡
患者作为数据真正的所有者，对"谁能看、看多久、看几次"几乎没有自主权。
Design Goals. 设计目标
  • 建立可验证、可审计、不可篡改的数据共享通道
  • 把访问控制策略从应用层下沉到链码层
  • 保留链下高效查询，链上保留唯一真相

VII.VERSION CHAIN  /  病历版本链

07previous_tx_id, threaded.

Append-only Semantics  /  设计要点

修订不是覆盖，而是 追加新版本 + 推进 LATEST 指针。每一版都带 TxID、共识时间戳和签名，第三方可独立验证。MySQL 仅存最新版以保证查询效率。

Query Strategy  /  查询路径

列表场景 → 直接查 MySQL 当前视图；溯源场景 → 调用 GetRecordHistory 触达 Fabric。Gateway 30 秒 TTL 缓存，写操作主动失效，避免重复 query peer。

VIII.FILE ENCRYPTION  /  文件加密

08GCM at rest, SHA-256 on chain.

UPLINK · 上传链路
Upload

DOWNLINK · 下载链路
Download
解密 + 双层校验
Why not the file?
为什么不上链原文
医疗文件大且敏感，账本不适合存原文；哈希上链已足够建立"曾经存在过"的证据。
Why GCM?
为什么是 GCM
CTR 模式吞吐高 + 自带 tag，密文被改立刻失败，无需额外 MAC。
Why double-hash?
为什么再算一次
GCM 防"密文被改"，链上哈希防"业务侧偷换底层文件"，两层针对不同攻击面。
Key custody?
密钥管理
主密钥经环境变量注入（MEDSHARE_FILE_KEY_BASE64），生产强制必填。

Four-attribute Policy
IX.ON-CHAIN ABAC  /  链上访问控制

09Four attributes, all enforced.

A.01
α
Subject MSP
主体身份
调用者必须来自申请时记录的组织，链码用 getMSPID 校验。冒充无效。
A.02
τ
Validity Period
有效期
expires_at 由患者审批时填入；到期链码自动拒绝，无需后端介入。
A.03
n
Read Counter
最大次数
remaining_reads 在 AccessRecord 中原子递减，归零即拒。
A.04
⊘
Explicit Revoke
显式撤销
RevokeAccess 写入 REVOKED；状态机不可回退，撤销即终态。

Why on-chain  /  为什么放在链码里

后端代码可能存在缺陷或被绕过；链码是所有 peer 节点共识的代码。每一次访问都是一次 transaction —— 天然审计；策略修改 = 链码升级 = 全网投票，无法静默变更。

Unauthorized Attempt  /  非法访问处理

链码触发 UnauthorizedAttempt 事件 → 经 Gateway 路由至后端事件总线 → 推送给患者与管理员 → 持久化到 audit_events 表，以备追责。

Data Mirror  /  链下镜像

四个属性同时镜像到 MySQL access_requests 表用于业务查询；但真相在链上 — 链下记录不一致时，以链码视图为准。

XII.SECURITY MODEL  /  纵深防御

12Six layers, stacked.

i.
传输层TRANSPORT
Nginx 强制 HTTPS，HSTS / X-Frame-Options / X-Content-Type-Options / Referrer-Policy；HTTP 自动 301 → 443。
ii.
认证层AUTHENTICATION
密码使用 bcrypt 哈希存储，登录签发 JWT，账号支持 is_active 软禁用。
iii.
授权层AUTHORIZATION
后端做角色粗粒度过滤；链码做细粒度 ABAC（MSP / 期限 / 次数 / 撤销）。
iv.
数据层DATA
文件 AES-256-GCM 加密落盘；链上仅存 SHA-256；主密钥来自环境变量。
v.
限流层RATE LIMIT
slowapi 按 IP 节流；登录 5/min、注册 10/min，缓解暴力破解与扫描。
vi.
审计层AUDIT
所有关键写操作均上链 + 落 audit_events 表；非法尝试同样可追踪。

XIV.SUMMARY  /  设计回顾

Three judgements,
four mechanisms.

Three Judgements  /  三个判断

  1. I.
    What goes on chain?
    什么上链 — 哈希、状态、策略；不上原文。
  2. II.
    Who decides?
    谁说了算 — 链码代码即规则，后端只是搬运工。
  3. III.
    How fast?
    如何保证响应 — MySQL 当前视图 + 30s 缓存 + 流式 IO。

Four Mechanisms  /  四个机制

M.01
病历版本链
previous_tx_id 串成单向链表，不可篡改的修订史。
M.02
双层完整性
AES-GCM tag + 链上 SHA-256，攻击面互不重叠。
M.03
链上 ABAC
期限 / 次数 / MSP / 撤销四属性同时生效。
M.04
事件驱动通知
setEvent → Gateway listener → WS 端到端推送。
Coda
把"医疗数据可信共享"从口号落地为一组可被独立验证的链上事实，同时保留了传统 Web 系统的开发效率与查询性能。
