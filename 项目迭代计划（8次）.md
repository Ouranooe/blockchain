**项目**：MedShare — 基于 Hyperledger Fabric 的医疗数据共享平台
**技术栈**：FastAPI + Vue3 + Node.js Fabric Gateway + MySQL + Hyperledger Fabric 2.x + Docker Compose

**迭代节奏**：每次 1-2 周，产出"代码 + 测试 + 一份实验报告"

## 迭代 1：工程基石 + 链码测试框架

**目标**：同时立起"后端测试"和"链码测试"两套自动化基础，并修复最严重的安全问题（密码明文）。

**功能完善**
- `POST /api/auth/register` 用户注册；`POST /api/auth/change-password` 改密
- 用户实体补字段 `msp_org`（Org1MSP / Org2MSP），登录时按医院映射对应组织身份

**区块链侧测试（重点）**
- 链码项目引入 **mocha + chai + sinon + fabric-contract-api 测试工具**
- 对现有 6 个链码方法编写单元测试：
  - 正常路径 / 重复创建冲突 / 未授权调用 / 非法状态跃迁
- 后端引入 pytest + httpx，覆盖 auth、register、改密

**优化**
- 密码使用 **bcrypt** 哈希 + 盐（passlib）
- JWT 密钥、DB 连接串从 `.env` 读取（替换硬编码）
- 链码测试命令：`npm test` 接入 CI-friendly 报告

**验证**
- `cd fabric-network/chaincode/medshare/javascript && npm test` 全通过
- `pytest backend/tests/ -v` 全通过
- 实验报告量化指标：链码测试覆盖率 ≥80%

## 迭代 2：病历版本链 + 链上修订追溯

**目标**：把"病历可修订"用链式结构在链上表达——每次修订上一笔新交易，指向前一版本的 txId，形成"版本链"。

**功能完善（链码改动为主）**
- 扩展链码方法：
  - `UpdateMedicalRecordEvidence(recordId, newHash)` — 创建新版本，写入 `previous_tx_id` 和 `version`
  - `GetRecordLatest(recordId)` / `GetRecordVersion(recordId, version)`
- 数据结构新增：`version`、`previous_tx_id`、`content_hash`、`updated_at`
- 后端提供病历"修订"接口；前端病历详情页显示版本树

**测试**
- 链码测试：
  - 连续修订 5 次，版本号正确递增
  - `previous_tx_id` 链条完整、每一版都指向前一版
  - 修订不存在的记录抛错
- 集成测试：后端 API 触发修订 → 链上版本数 +1

**优化**
- 键设计：`RECORD_{id}_v{version}`（避免覆盖）+ `RECORD_LATEST_{id}`（热点索引）
- 查询历史版本时只读热点索引 + 按需回溯
- 前端版本树懒加载

**验证**
- 实验报告量化：
  - 修订 10 次，链上可回溯完整链条
  - 查询最新版延迟 < 查询全部历史 × 0.2

## 迭代 3：Fabric 历史查询与链上时间线

**目标**：利用 Fabric 原生的 **`GetHistoryForKey`** API，在不维护链下表的前提下，还原一条病历的完整历史轨迹。

**功能完善（链码）**
- 新增链码方法 `GetRecordHistory(recordId)`：调用 `ctx.stub.getHistoryForKey()`，返回 `[{txId, timestamp, value, isDelete}...]`
- 新增 `GetAccessRequestHistory(reqId)`：审批流完整状态轨迹
- 后端封装 `/api/records/{id}/history`；前端病历详情页新增"**链上时间线**"组件

**测试**
- 链码测试：
  - mock `getHistoryForKey` 返回多条记录，验证解析
  - 历史条目按时间倒序
- 集成测试（实跑链）：
  - 上传 → 修订 3 次 → 查询历史得 4 条
  - 每条 txId 可通过 `peer chaincode query` 独立验证

**优化**
- 历史结果在网关层加 **TTL 缓存**（30s），同一病历 30s 内重复查询命中缓存
- 前端时间线按页加载（每页 20 条）

**验证**
- 区块链侧量化指标：
  - 链上可回溯率 100%（链码无需额外辅助表）
  - 100 次历史查询缓存命中率 ≥85%

## 迭代 4：文件哈希上链 + 链下加密存储

**目标**：真实承载医疗文件（PDF/图片），体现区块链"**链上存证、链下存文件**"的经典范式。

**功能完善**
- `POST /api/records` 支持 multipart 文件上传（PDF/JPG/PNG，≤10MB）
- 服务端 **AES-256-GCM** 加密存储到 `backend/storage/`
- 计算文件 **SHA-256** → 上链；链上只存 `content_hash`，链下存密文
- 下载接口：校验权限 → 读取密文解密 → **重新计算哈希 → 与链上对比** → 不一致则拒绝

**测试**
- 单元测试：
  - 加密/解密往返：明文一致
  - 篡改密文后下载校验失败（哈希不一致）
  - 篡改链上哈希（通过直连 peer）后下载报警
- 集成测试：5MB PDF 上传下载，链上哈希 == 本地 SHA-256

**优化**
- 流式加密（避免整文件加载到内存）
- 下载支持 Range Header（断点续传）
- 前端显示 "链上哈希 ✓" / "文件完整性 ✓" 双勾

**验证**
- 量化指标：
  - 篡改密文 → 100% 检出
  - 篡改链上哈希 → 100% 检出
  - 10MB 文件加密吞吐 ≥ 30MB/s

## 迭代 5：链上访问控制精细化（ABAC）

**目标**：把"有期限授权 / 链上撤销 / 访问次数限制"的策略**写进链码**，而不是靠后端代码守卫——这是区块链"不可绕过"的核心价值。

**功能完善（链码改动为主）**
- `ApproveAccessRequest` 接收 `duration_days` 和 `max_reads`，写入 `expires_at` 和 `remaining_reads`
- 新增 `AccessRecord(recordId, requesterId)` 链码方法：
  - 校验存在已批准的有效请求（未过期、剩余次数 > 0）
  - 命中则 `remaining_reads--` 并记录一笔访问事件
  - 不命中则拒绝并返回原因
- 新增 `RevokeAccessRequest(reqId, patientId)`：链上撤销（校验调用方身份 == 患者）
- 利用 Fabric **ClientIdentity**（`ctx.clientIdentity.getAttributeValue` 或 MSP ID）做调用方校验

**测试**
- 链码测试（关键）：
  - 过期授权访问被拒
  - 次数用尽后访问被拒
  - 非患者本人无法撤销
  - 撤销后立即访问被拒
  - 状态机合法跃迁表驱动测试
- 集成测试：医院 A 访问医院 B 的未授权病历直接被**链码层**拒绝（而非后端）

**优化**
- 链码方法内 `getState` 次数最小化（合并读取）
- 过期判断用 `ctx.stub.getTxTimestamp()`（权威时间，而非链下时间）

**验证**
- 量化指标：
  - 即使绕过后端直接调网关，非法访问 100% 被链码拒绝
  - 链码方法平均 GetState 次数 ≤3

## 迭代 6：链码事件 + 实时审计告警

**目标**：用 Fabric **Chaincode Events** 打通"链上发生事件 → 链下即时响应"的通路，实现实时推送与异步审计。

**功能完善（链码 + 后端）**
- 链码在关键操作中调用 `ctx.stub.setEvent(eventName, payload)`：
  - `RecordCreated` / `RecordUpdated` / `AccessApproved` / `AccessRevoked` / `UnauthorizedAttempt`
- 网关层（Node.js）使用 `contract.addContractListener()` 订阅事件
- 后端订阅收到事件 → 写入 AuditLog + 通过 **WebSocket** 推送到前端（例：患者即时收到"你的病历被访问"）
- 前端右上角消息铃铛 + 审计页实时刷新

**测试**
- 链码测试：每个操作触发对应事件、payload 正确
- 集成测试：上链操作 → 事件订阅方收到 → WebSocket 推送到前端（E2E）
- 异常测试：订阅断开后自动重连、事件不丢失（offset 管理）

**优化**
- 订阅 offset 持久化（断线恢复从上次位置继续，避免漏 / 重）
- 后端订阅改为独立后台任务（避免阻塞主请求）
- 高频事件批量落库（每 100 条或 1s flush）

**验证**
- 量化指标：
  - 上链 → 前端通知 **端到端延迟 P95 < 2s**
  - 断线重启后 0 事件丢失

## 迭代 7：CouchDB 富查询 + 链上条件检索

**目标**：将 Fabric 世界状态数据库由 LevelDB 切到 **CouchDB**，启用富查询（Rich Query），支持"按医院 / 按时间段 / 按分类"的链上条件检索。

**功能完善**
- 修改 [docker-compose.yml](docker-compose.yml) 或 Fabric 启动脚本，把 peer 状态库切到 CouchDB
- 链码新增：
  - `QueryRecordsByHospital(hospitalId, pageSize, bookmark)` — 用 `getQueryResultWithPagination`
  - `QueryRecordsByDateRange(fromTs, toTs)`
  - `QueryPendingRequestsForPatient(patientId)`
- 为常用字段创建 CouchDB 索引（`META-INF/statedb/couchdb/indexes/*.json`）
- 后端替换部分原本走 MySQL 的列表接口，改为链上查询（源头真实性更高）

**测试**
- 链码测试：富查询结果正确（注意 mock CouchDB 响应）
- 集成测试：插入 1000 条病历，按医院/时间条件查询结果准确
- 对比测试：无索引 vs 有索引，同一查询耗时对比

**优化**
- 索引文件合理设计（避免全表扫描）
- 分页用 bookmark 而非 offset
- 高频查询结果加网关缓存

**验证**
- 量化指标：
  - 1000 条记录下，有索引查询 < 无索引查询 × 0.3
  - 分页遍历 1000 条无丢失 / 重复

## 迭代 8：性能压测（Caliper）+ 生产化部署

**目标**：用专业工具量化系统的链上 TPS 与延迟，并完成生产化部署闭环。

**功能完善**
- 编写 **Hyperledger Caliper** 测试配置：
  - 场景 1：纯上链（CreateRecord）
  - 场景 2：纯查询（GetRecord）
  - 场景 3：混合读写（80% 查 + 20% 写）
  - 输出 Caliper 标准报告（Throughput、Avg Latency、P95、错误率）

- MySQL 自动备份脚本（每日 dump，保留 7 天）

**测试**
- **链 TPS 压测**（Caliper）：报告 TPS / 延迟 / 错误率曲线

- **安全扫描**：`bandit backend/` + `npm audit` + OWASP ZAP 基础扫描

  - Caliper 场景 2（查询）TPS ≥ 200
  - Caliper 场景 1（上链）TPS ≥ 30（受 Fabric 出块限制）

## 风险与备选方案

- **Fabric 学习曲线陡**：迭代 5（ABAC）如果 ClientIdentity 属性配置复杂，可先用 MSP ID 做粗粒度校验
- **CouchDB 切换影响大**：迭代 7 需要 teardown 重建 Fabric 网络；建议保留切换前快照便于回滚
- **Caliper 环境复杂**：迭代 8 如时间紧张，可用简化版脚本（并发 goroutine/async 调网关）替代
- **链码升级**：迭代 2/3/5/6/7 均涉及链码升级，需熟悉 `peer lifecycle chaincode upgrade` 流程
